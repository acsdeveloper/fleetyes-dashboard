🚨 Issues Requiring Action

Critical — Security

XSS Stored Raw — All Text Fields

Submitting <script>alert(1)</script> as the value of document_type, category name, or compliance doc title results in the raw HTML being stored and returned in API responses. Any UI rendering these fields without additional escaping is vulnerable to stored XSS.
Fix: Apply strip_tags() or an HTML purifier on input in the relevant model mutators or FormRequest validation rules. Alternatively, validate that these fields contain no HTML tags.
Critical — Stability

500 Server Error on Invalid documenttype_id

Sending an all-zeros UUID as documenttype_id on POST /compliance-documents triggers an unhandled Illuminate\Database\QueryException with a full HTML stack trace. This exposes internal database query structure.
Fix: Add 'documenttype_id' => 'exists:compliance_document_types,uuid' to the FormRequest validation rules so that invalid UUIDs return 422 before the query runs.
High — Missing Validation

Required Fields Not Validated — POST Endpoints

Multiple POST endpoints return 200 {"message":"Fleetbase API"} instead of 422 when required fields are omitted or invalid. This means requests silently fail — callers have no way to detect the error. Affected fields:
  • /compliance-document-types — missing entity_type, document_type, company_uuid; invalid entity_type value; invalid company_uuid
  • /compliance-business-categories — missing name, empty name, missing company_uuid
  • /compliance-documents — missing entity_type, company_uuid; invalid vehicle_uuid (creates orphaned record!); wrong UUID field (driver_uuid on vehicle entity); invalid expires_at format; notes over 500 chars; business doc with documenttype_id instead of category_id; category_id=other with no custom_category_name
Fix: Add Laravel FormRequest classes for each POST endpoint with required, in:vehicle,driver, exists:, date, and max: rules. The 200 fallback response suggests requests are hitting Fleetbase's base route instead of the intended controller — verify route registration.
High — Data Integrity

Invalid vehicle_uuid Creates Orphaned Record

POST /compliance-documents with vehicle_uuid: "00000000-0000-0000-0000-000000000000" (a fake UUID) returned 201 Created with a real document UUID. A compliance document now exists linked to a non-existent vehicle. This will cause UI crashes when trying to load the associated entity.
Fix: Add 'vehicle_uuid' => 'exists:vehicles,uuid' and 'driver_uuid' => 'exists:drivers,uuid' to FormRequest validation (required when entity_type matches). The orphaned record ef896dbf-eaaa-45e2-96ec-8f1339a6a516 should be cleaned up.
High — Missing Validation

String Length Limits Not Enforced

Fields documented with max 191 characters (document_type, category name) and max 500 characters (notes) accept strings of 300+ / 600+ characters respectively without error. This can cause database truncation silently or exceed column limits.
Fix: Add max:191 and max:500 validation rules in FormRequest. Alternatively, apply DB-level column constraints.
Low — RFC 7235

Unauthenticated Requests Return 400 Instead of 401

All API endpoints return 400 {"errors":["Unauthenticated."]} when no token is provided. Per RFC 7235, the correct status code for missing/invalid credentials is 401 Unauthorized. Some HTTP clients use 401 to trigger re-authentication flows.
Fix: In the Authenticate middleware, change the response status to 401 instead of 400.
Low — Cosmetic

DELETE /walkaround-checks/{fake_uuid} Returns 405

Attempting to DELETE a walkaround check with a non-existent UUID returns 405 Method Not Allowed instead of 404 Not Found. This suggests the DELETE route isn't registered for this resource or the route pattern doesn't match.
Fix: Verify that Route::delete('/walkaround-checks/{id}', ...) is registered, or add a 404 check before the 405 response.
Low — Push Config

POST /test-notification with channel=push returns 400

Sending a test push notification returns 400 with no descriptive error message. This is expected when no mobile device is registered for the account, but the generic 400 with empty body gives callers no way to distinguish "no device registered" from a real error.
Fix: Return a descriptive error: {"success": false, "message": "No push device registered for this user", "errors": ["..."]} so callers can surface a helpful message in the UI.
