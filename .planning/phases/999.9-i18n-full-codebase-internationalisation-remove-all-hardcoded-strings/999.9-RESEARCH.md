## Implementation Constraints

1. **Type safety**: The `Translations` type must be updated in SYNC with the translations object. Add to type first, then add to all 6 language objects.
2. **No key renames**: Only additive changes — existing `t.xxx` references in working pages MUST NOT break.
3. **useLang import**: Pages without `useLang` need `import { useLang } from "@/components/lang-context"` added, and `const { t } = useLang()` in each component function.
4. **Non-English translations**: Use reasonable machine translations for de/fr/es/it/pl — accuracy matters more than fluency for this pass.
5. **API-level strings**: Values like `"pending"`, `"processed"`, `"failed"` used in API calls stay hardcoded — only translate at display-time.
6. **Dynamic strings**: Strings with interpolated values (e.g. `"Zipping ${n} files…"`) need template-friendly approach — keep as functions or use `{n}` placeholder pattern already used in lang-context.
7. **Toast/alert strings**: Check `toast()` calls with hardcoded strings — route through `t`.

## Wave Strategy

Given scope, recommend 4 waves:

**Wave 1 (Parallel):** Extend lang-context.tsx with all new keys for ALL namespaces + all 6 languages. Single plan, foundational — everything else depends on it.

**Wave 2 (Parallel):** Replace hardcoded strings in HIGH-priority pages (compliance, toll, fuel, trips) — these have most strings and are already partially using `t`.

**Wave 3 (Parallel):** Replace hardcoded strings in MEDIUM + LOW priority pages (fleet-management, import-hub, drivers, holidays, parking, org-settings, off-shift, fleets, vehicles, places, maintenance-*, calendar, rota, settings, issues, inventory).

**Wave 4:** TypeScript compilation check + final grep audit for any remaining hardcoded strings. Create verification evidence.
