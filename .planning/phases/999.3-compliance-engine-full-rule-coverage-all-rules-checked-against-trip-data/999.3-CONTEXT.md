**Gathered:** 2026-04-01

<domain>
## Phase Boundary

Every rule displayed in the compliance panel must actually fire when violated.
The engine must correctly handle all data states (full data, partial data, no data).
New tests must cover every rule. Nothing is deployed until all tests pass.

This phase covers:
1. Fix any rules that silently skip non-driving (NON_DRIVING_DUTY) working days
2. Remove the MISSING_END_TIME hard block from the prospective check
3. Flag trips with missing end times in the Trips list UI (not at assignment time)
4. Add a `breaksAssumedCompliant` configuration flag
5. Fill all test gaps (weekly rest, fortnightly, reduced rest limit, no-data/clean-slate)
6. Verify all 35 existing tests still pass after changes
7. Deploy only once the full suite is green

</domain>

<decisions>
## Implementation Decisions

### A тАФ No-Data / New Business Behaviour
- **D-01:** No trip history = driver was resting. Current adapter behaviour (`buildRestDay()` for days with no trips and no rota entry) is **confirmed correct**. Do NOT change it.
- **D-02:** The compliance panel for a clean-slate driver should show no violations and no warnings. The 29-day record-keeping warning SHOULD appear (it's factually correct тАФ there are no records) but this is already implemented in `record-keeping.ts` and does not need to change.
- **D-03:** Tests must include a "clean-slate driver" scenario: 29 days of no data тЖТ zero violations, zero warnings (record-keeping soft warning is acceptable; hard violations must be zero).

### B тАФ Missing End Time: Remove Hard Block, Flag in Trips List
- **D-04:** Remove the `MISSING_END_TIME` hard-violation block from `prospective-check.ts`. A trip with no `estimated_end_date` and no `time` field should **not block assignment**. The 2h fallback is already in place and is acceptable for gap calculation purposes.
- **D-05:** The Trips list page (`apps/v4/app/(dashboard)/trips/page.tsx`) must visually flag any trip record that is missing `estimated_end_date` AND `time` as a **"Risk: No end time"** indicator. Style it as a yellow/amber warning badge or icon inline in the row. This is where the user will notice and fix data quality issues before assigning.
- **D-06:** The existing test "blocks assignment when estimated_end_date is NULL and time is 0" (prospective-rest-gap.test.ts line 65) must be **updated** to reflect the new behaviour: no MISSING_END_TIME violation, gap check uses 2h fallback.

### C тАФ "Working Time" Definition & Break Assumption Flag
- **D-07:** For this implementation, all time between `scheduled_at` (trip start) and `estimated_end_date` / `time`-derived end is treated as **total working time** (NON_DRIVING_DUTY). This applies to both the adapter and the prospective check.
- **D-08:** We **assume breaks are taken within the working window** in compliance with law. This means the break requirement check (`EU_BREAK_REQUIREMENT` in the per-day rules of `assimilated.ts`) only fires for `DRIVING`-type activities. For `NON_DRIVING_DUTY`, breaks are considered self-managed and compliant.
- **D-09:** Add a `breaksAssumedCompliant: boolean` flag to the engine config/vehicle config type. Default: `true` (current behaviour). When `false` (future mode), the break check will also fire for NON_DRIVING_DUTY days. For this phase: implement the flag and wire it up, but default it to `true` so no behaviour changes yet.
- **D-10 (Critical Fix):** The batch validator (`assimilated.ts`) currently guards the per-day rules with `if (!day.hasDriving) continue`. Since all existing trips are `NON_DRIVING_DUTY`, `hasDriving` is always `false`, meaning the **daily working hours limit (EU_DAILY_WORK_LIMIT) never fires in the batch validator**. This guard must be changed to `if (day.isRestDay) continue` so that working days with NON_DRIVING_DUTY activities are also checked against the 13h/15h daily limit. The break check within the same block must remain gated on `hasDriving` (per D-08 above).

### D тАФ Test Coverage Requirements
- **D-11:** Tests must verify all 8 rules listed in the compliance panel. Currently missing:
  - **Weekly rest (45h / 24h reduced):** passing case + violation + consecutive-reduced violation
  - **Fortnightly driving limit (90h):** clean + violation + warning
  - **Reduced daily rest limit (3├Ч/week cap):** warning at 3rd, violation at 4th
  - **Daily working hours in batch validator (NON_DRIVING_DUTY days):** violation at >13h, warning at >13h, violation at >15h тАФ must fire now that D-10 is applied
  - **Clean-slate driver (no data, 29 days):** zero hard violations
  - **Prospective check: missing end time no longer blocks:** test updated per D-06
- **D-12:** All 35 currently-passing tests must still pass after every change.
- **D-13:** No deployment (git push / production build) until `npx vitest run apps/v4/lib/compliance-engine` exits with 0 failures.

### Agent's Discretion
- Whether to add `breaksAssumedCompliant` to `VehicleConfig` or as a separate engine-level option
- Exact styling of the "Risk: No end time" badge in the Trips list (amber, consistent with existing trip status badges)
- Whether to add a `no-data` test as a new file or append to an existing test file

</decisions>

<specifics>
## Specific Ideas

- "All time between start and end = working time" is the explicit current model. Breaks are self-managed by the driver within that window. The compliance engine trusts compliance with break law.
- `breaksAssumedCompliant` flag: implement now as `true`, don't change behaviour yet. This is a forward-compatibility hook.
- The "Risk: No end time" flag in Trips list is a **data quality indicator**, not a compliance violation. It should be visible without navigating to the compliance panel.
- User's priority: if the data exists, check everything. Zero tolerance for rules that silently skip.

</specifics>
