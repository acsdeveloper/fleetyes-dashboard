# Phase 999.4: Compliance Engine ‚Äî Rule Logic Rethink ‚Äî Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

The compliance engine's violations are correct in the test suite but fail silently in the UI
because of a date-matching problem: the rota board filters violations by `v.date === date`
(exact day match), but several rules emit violations with a week-start date rather than the
actual working day. The result: many real violations are invisible in the cell view.

This phase fixes the violation attribution chain end-to-end:

1. Make every rule emit violations with the exact working-day date where the issue occurred
2. Enrich every violation with the specific trip(s) that caused it (tripId, start, end)
3. Remove diagnostic console.warn spam from prospective-check.ts
4. Ensure the compliance panel can render violation context (trip details) when present

This phase does NOT restructure the engine architecture. The rule-loop is correct.
The data flowing out of each rule needs to be richer and better dated.

</domain>

<decisions>
## Implementation Decisions

### A ‚Äî Violation Date Attribution (The Core Bug)

- **D-01:** The root cause is that aggregate-window rules (weekly driving, fortnightly,
  weekly rest, consecutive days) emit their violation with `date: ws` (the ISO week Monday)
  rather than the specific working day that pushed the total over the limit. The rota board
  filters by exact date match (`v.date === date`), so these violations are invisible on cells.

- **D-02:** Fix: every violation that is triggered by a specific working day MUST use that
  day's date as `date`. For aggregate rules:
    - Weekly driving: use the date of the LAST day in the week that caused the limit to be crossed
    - Fortnightly driving: same ‚Äî use the last day that crossed the 90h cap
    - Weekly rest: use the date of the first working day in the rest-deficient week
      (since the problem is "driver worked this day without adequate rest before/after")
    - Consecutive days: ALREADY correct ‚Äî uses `day.date` (the day that broke the streak)
    - Daily rest: ALREADY correct ‚Äî uses `curr.date`

- **D-03:** Do NOT change `getCellCompliance` in `rota/page.tsx` ‚Äî date-exact matching is
  correct once the engine emits the right dates. The UI is not the bug.

### B ‚Äî Trip Context on Violations

- **D-04:** Add optional trip context fields to `ComplianceViolation` in `types.ts`:
  ```ts
  tripId?:        string   // UUID of the triggering order/trip
  tripStartTime?: string   // ISO datetime of trip start
  tripEndTime?:   string   // ISO datetime of trip end (or best estimate)
  ```
  These are OPTIONAL (undefined for rules that don't have a single triggering trip).
  Keep existing fields unchanged ‚Äî this is backwards-safe.

- **D-05:** Which rules get trip context:
    - `EU_DAILY_REST` (batch validator): set `tripId` to the trip on `curr` day that
      caused the short rest. Use the first activity on `curr` as the arriving trip.
    - `TRIP_OVERLAP`: set `tripId` to the UUID of the overlapping activity (if available
      from the `Activity` interface ‚Äî check if `orderId`/`tripId` is on Activity first;
      if not, leave undefined for now).
    - `EU_DAILY_WORK_LIMIT` (NON_DRIVING_DUTY): set `tripId` to the trip on that working
      day (if only one trip that day; if multiple, leave undefined).
    - Aggregate rules (weekly, fortnightly, consecutive): leave `tripId` undefined ‚Äî
      these span multiple trips and no single trip is "the" cause.
    - Prospective check violations: ALREADY has the trip ‚Äî the `newTrip.uuid` IS the
      triggering trip. Set `tripId: newTrip.uuid`, `tripStartTime`, `tripEndTime` on
      all prospective violations.

- **D-06:** The `Activity` interface in `types.ts` needs `orderId?: string` added so the
  adapter can tag each activity with its source Order UUID. Without this, the batch
  validator cannot know which trip an activity came from. The adapter (`orderToActivity`)
  should set `orderId: order.uuid`.

### C ‚Äî Diagnostic Noise

- **D-07:** Remove ALL `console.warn` blocks from `prospective-check.ts`. There are
  3 warn blocks (at lines ~193‚Äì199, ~229‚Äì231, ~310‚Äì314). Remove them entirely ‚Äî
  production should be silent. Tests pass without them.

### D ‚Äî UI Rendering of Trip Context

- **D-08:** The compliance panel in `rota/page.tsx` should render `tripId`,
  `tripStartTime`, and `tripEndTime` when present on a violation. Display format:
  inline below the violation message as a subdued detail row:
  `Trip ¬∑ {tripStartTime formatted} ‚Üí {tripEndTime formatted}`
  Keep it minimal ‚Äî no new components needed, just conditional rendering within
  the existing violation card.

- **D-09:** No structural UI changes to the compliance panel. Just add the trip detail
  row below `message` and `calculation`.

### E ‚Äî Test Coverage

- **D-10:** Add targeted regression tests for the date attribution fix:
    - Weekly driving violation ‚Üí `v.date` equals the last day of the violation week, not Monday
    - Fortnightly violation ‚Üí same pattern
    - Weekly rest violation ‚Üí `v.date` equals first working day of the rest-deficient week
  These tests catch the exact bug described in D-01 and prevent regression.

- **D-11:** All existing 71 tests must still pass. No test-breaking changes.

- **D-12:** No deployment until `npx vitest run apps/v4/lib/compliance-engine` exits 0.

### Agent's Discretion
- Exact format of `tripStartTime`/`tripEndTime` in violation detail row (ISO or locale string)
- Whether to use `toLocaleString` or `toISOString` for display ‚Äî locale string preferred
- Whether `orderId` on Activity is named `orderId` or `tripId` ‚Äî use `orderId` to avoid
  conflict with the violation field name

</decisions>

### How getCellCompliance Works (rota/page.tsx L883‚Äì893)
```ts
function getCellCompliance(driverUuid: string, date: string) {
  const report = complianceReports.get(driverUuid)
  return {
    violations: report.violations.filter(v => v.date === date),  // ‚Üê EXACT MATCH
    warnings:   report.warnings.filter(w => w.date === date),
  }
}
```
This is correct. Violations must carry the right `date` to be visible per-cell.

<specifics>
## Specific Ideas

- The weekly driving loop already tracks `weekDays` ‚Äî the last day over the limit is
  `weekDays[weekDays.length - 1].date` (assuming sorted, which they are).
- For fortnightly: same pattern with `fortDays`.
- For weekly rest: the first working day of the week is `weekDays.find(d => !d.isRestDay)?.date ?? ws`.
- The `console.warn` spam is the exact output being seen in browser devtools ‚Äî removing it
  will clean up production immediately.
- User confirmed: violations are surfacing in tests but not in the UI cell view. The date
  mismatch is the confirmed mechanism.

</specifics>
