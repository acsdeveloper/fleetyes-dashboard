# Phase 999.1: Rota Compliance Engine — Fix & Verify — Context

**Gathered:** 2026-03-31

<domain>
## Phase Boundary

Make the rota compliance engine work correctly. Every EU/Assimilated rule that is already coded must actually fire when violated, and every rule must be backed by unit tests that prove it works.

The engine already has all the rules implemented in code. The problem is confidence — the user has no proof any of it works. Overlapping trips (the most basic check) is failing, which undermines trust in every other rule.

This phase does NOT add new rules. It fixes, verifies, and hardens the existing implementation.

</domain>

<decisions>
## Implementation Decisions

### Enforcement Strategy
- **D-01:** Defense in depth — overlaps and all violations must be caught at BOTH enforcement points:
  1. **Prospective check** (synchronous, at drag-drop/popover time) — blocks the allocation
  2. **Batch validator** (background scan via `validateAssimilated()` / `validateGBDomesticGoods()`) — catches anything that slips past
- **D-02:** Zero-gap (back-to-back trips) is allowed. Only negative gaps (actual time overlap) are violations.

### Rule Priority (verification order)
- **D-03:** All existing EU/Assimilated rules must work. Priority order for fix & verify:
  1. **Overlapping trips** — same driver, two simultaneous trips → hard violation
  2. **Daily rest** — 11h between working days (9h reduced, max 3×/week)
  3. **Daily working hours** — 13h max (15h with reduced rest)
  4. **Weekly/fortnightly driving limits** — 56h/week, 90h/fortnight
  5. **Consecutive working days** — max 6 before weekly rest required
  6. **Weekly rest** — 45h (24h reduced, alternating weeks)
  7. **Record keeping** — 29-day rolling window coverage

### Race Condition Fix
- **D-04:** Harden localStorage `trip_data` as the canonical source for the prospective check. It's synchronous (written before async API call). The `confirmReassign` path must also write `trip_data` (currently missing).

### Test Strategy
- **D-05:** Unit tests for every rule in the compliance engine. Each rule gets at least:
  - A passing case (no violation)
  - A violation case (triggers hard violation)
  - A warning case (approaching limit)
  - Edge cases specific to the rule (midnight crossing, tramper exemption, DST)

### Agent's Discretion
- Test framework choice (vitest already configured in project root)
- Test file organization within compliance-engine directory
- Whether to refactor adapter.ts to deduplicate overlapping activities or just detect them

</decisions>

<code_context>
## Existing Code Insights

### Known Bugs (from audit)
1. **Race condition**: Fast consecutive drops bypass overlap check — `tripIndex` is React state (async), `trip_data` in localStorage is synchronous but `confirmReassign` doesn't write it
2. **No batch overlap detection**: `assimilated.ts` has zero overlap checks — if one slips past prospective, it's never caught
3. **Duration double-counting**: `buildWorkingDayFromTrips()` pushes overlapping activities without merging — sums their durations independently
4. **No cross-day overlap check in batch**: Trips spanning midnight could overlap with next day's trip

### Reusable Assets
- `vitest.config.ts` already configured at project root
- `utils.ts` has all the time-math helpers needed (`activityMinutes`, `restBetweenDays`, `fmtMinutes`)
- `prospective-check.ts` has working overlap logic (lines 198-208) that can be extracted for reuse in batch

### Established Patterns
- All validators return `ComplianceViolation[]` — consistent interface
- Rule constants are clearly defined at top of each validator file
- Activities use `Date` objects for `startTime`/`endTime` — time comparison is straightforward

### Integration Points
- Rota page calls `prospectiveComplianceCheck()` at 3 points: handleSave (L948), handleDrop (L1093), confirmReassign (L1143)
- Batch validation runs via `runComplianceChecks()` (L851) after every allocation change
- `buildDriverRecord()` in adapter.ts is the bridge between API data and engine types

</code_context>
